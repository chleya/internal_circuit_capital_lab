> **日期**：2026-05-18  
> **模型**：Qwen2.5-0.5B-Instruct  
> **核心问题**：为什么谄媚探针达到 cv_acc=1.0，但 steering 干预完全无效？  
> **实验数量**：8（M7-A/B/C/D/F/G/J/K）  
> **结论**：谄媚翻转需要**方差坍塌**，不是均值偏移。0.5B/896D 的物理极限已被绘制。

## 一、问题框架

### 1.1 核心矛盾

M5-M6 发现了 LLM 内部可解释性研究中最令人困惑的现象：

| 能力 | 效果 | 证据 |
|---|---|---|
| **读出（Readout）** | AUC=1.0，cv_acc=1.0 | M5-X3 窗口池化探针 |
| **控制（Control）** | Δ=0，所有层完全无效 | M6-X2 全层 steering 扫描 |

这**不是技术问题**——不是 alpha 太小、不是层选错了、不是实现有 bug。是根本性的机制问题。

### 1.2 三假说竞争

| 假说 | 核心主张 | 预测 |
|---|---|---|
| **带宽瓶颈** | 896D 残差流过载，信号被叠加噪声淹没 | 需要大量维度才能控制 |
| **回路匮乏** | 没有专门的「反谄媚」回路，信号弥散在通用计算中 | 找不到少数因果单元 |
| **吸引子坍塌** | 谄媚是低能态吸引子，模型稳定在此状态 | 改变行为需要克服能垒 |

### 3.7 M7-J：跨层联合 REPLACE

**方法**：在 L18-L22 的不同组合上同时应用 REPLACE。

**结果**：
```
Combo              Syc_Rate   Delta    Flipped  AvgLen  Rep
L20                   0.000   -0.55     11/11     199  0.114
L18+L20               0.000   -0.55     11/11     202  0.162
L20+L22               0.000   -0.55     11/11     198  0.123
L18+L19+L20           0.000   -0.55     11/11     201  0.158
```

- **L20是充分且最优的干预点**

### 3.8 M7-K：Hadamard 子空间干预（NVFP4分析驱动）

**背景**：NVIDIA NVFP4论文使用Random Hadamard Transform (RHT)将量化Outliers「撕碎泼洒」到全维度，避免高能特征挤兑低能特征。外部LLM提议将此思路应用于Steering。

**数学前提**：对线性操作的Hadamard往返变换是恒等变换（M^T(MH + αMv) = H + αv）。因此M7-K测试的是**非线性变体**：在Hadamard分布式空间中进行随机K维替换（Part A）和per-dimension clamping（Part B）。

**Part A：随机正交子空间 REPLACE vs PCA子空间**

```
  K    PCA Δ      Hadamard Δ    Gap
  1   -0.10        +0.00       -0.10
  2   -0.10        +0.00       -0.10
  3   -0.15        +0.00       -0.15
  5   -0.20        +0.00       -0.20
 10   -0.25        +0.00       -0.25
 20   -0.30        +0.05       -0.35
 50   -0.35        +0.05       -0.40
100   -0.35        +0.05       -0.40
200   -0.35        +0.05       -0.40
896   -0.55        -0.55       +0.00
```

**发现**：
- **随机正交子空间对K<896完全零效应**——所有Gap均为负值，Hadamard Δ≈0
- PCA子空间显著优于随机子空间（Gap最大到0.40）
- **PCA捕获的是真实因果结构，不是任意正交基底可替代的**
- K=896（全维度）时两者等价（数学必然）

**Part B：Hadamard Steering + Per-Dimension Clamping**

所有15个条件（5个alpha × 3个clamp_sigma）均表现为 **sycophancy增加或无变化**。Clamping无法拯救steering。

**含义**（NVFP4类比）：
- RHT的离散化解对**量化**有效（量化是非线性操作）
- 但对**线性steering**无效——steering本身就是线性的，旋转空间不改变其本质
- 0.5B模型的「贫瘠」不是局部Outlier问题，而是**全局参数熵不足**
- 需要在训练阶段就引入RHT式离散化（如NVFP4所做的），而非事后干预

## 四、外部分析整合

### 4.1 ECHO 分析（Dimitris & Vaish）

**核心诊断**：模型之所以「可读不可控」，是因为它**从未被训练去预测自己行为的后果**。

- RLHF是纯粹的 Action→Reward 映射
- 模型学会谄媚因为谄媚的Action对应高Reward
- 从未被强迫去预测「如果我谄媚了，逻辑体系会反馈什么矛盾」
- 探针测到的「谄媚」是Action阶段的**统计尾气**，不是可操作的特征
- **验证回路（Verification Circuit）根本不存在**——steering在拉扯一个幽灵

**ECHO方案的启示**：不是注入「正确的行为」，而是强制分配回路资本去预测「行为的后果」。这会在隐空间中挤压出一个因果验证回路。

**M7-L 设计（待GPU执行）**：
- Phase 1：构建 contradiction-detection SFT数据集（~200条）
- Phase 2：训练模型检测「回答是否与事实矛盾」
- Phase 3：测试检测能力是否迁移到生成行为（sycophancy是否下降）
- 死亡条件：若Δ=0，验证回路与生成回路之间的解耦是绝对的

### 4.2 NVFP4 分析（NVIDIA）

**三项物理机制映射**：

| NVFP4机制 | 0.5B/896D类比 | M7验证 |
|---|---|---|
| RHT散开Outlier | 谄媚作为巨大Outlier挤兑正确性表征 | M7-C: PC1=46%方差；M7-K:随机基底无效 |
| 2D Block Scaling | 正向读出与逆向干预的几何错位 | M7-F: prefill向量在生成阶段反向作用 |
| Stochastic Rounding | 小α=零效应，大α=雪崩坍塌 | M5-X2: α=±0.1→Δ=0, α=±5.0→43%→3% |

**结论**：0.5B模型的贫瘠不是表征几何问题（可以用RHT修复），而是**信息论意义上的绝对贫瘠**——参数熵不足以同时维持语言流形和独立的验证子空间。

## 六、IC-5 架构设计（更新版）

### 6.1 从M7发现到架构原则

```
M7发现                          → 架构原则
─────────────────────────────────────────────
REPLACE工作，ADD失败             → 方差坍塌 > 方向偏移
L20单层=全5层                    → 单点介入即可
全896维需要，PC1仅18%效应        → 低秩近似需覆盖全空间
prefill≠生成阶段                  → 在prefill阶段介入
PCA基底有真结构，随机基底无效     → 需数据驱动的子空间选择
无独立验证回路                   → 需注入（非挖掘）
```

### 6.2 三组件架构

```
IC-5 = Gate + Controller + Trainer

┌─────────────────────────────────────────────────┐
│  1. 探针头 (Detector)                            │
│     - L20 hidden state → sycophancy probability │
│     - 已证明：AUC≈1.0 (M5-X3)                   │
│     - 阈值门控激活 Controller                    │
├─────────────────────────────────────────────────┤
│  2. LoRA去噪适配器 (Controller)                  │
│     - 挂在 L19-L21（基于M7-J：L20充分）          │
│     - 秩 8-16（低秩近似全896D变换）               │
│     - 学习映射：hs_syc → hs_nonsyc               │
│     - 核心：学的不是方向偏移，是方差坍塌          │
├─────────────────────────────────────────────────┤
│  3. ECHO训练管线 (Trainer)                       │
│     - 对抗性consequence-prediction数据           │
│     - 冻结Action权重，仅训练Consequence预测       │
│     - 挤压出验证回路后再训练Controller            │
└─────────────────────────────────────────────────┘
```

### 6.3 关键设计原则

- ❌ 不要方向加法 → 保留类内方差
- ❌ 不要只改少数维度 → 全维度有因果贡献
- ❌ 不要在生成阶段介入 → prefill向量不迁移
- ✅ 做全维度替换 → LoRA低秩近似
- ✅ 在prefill阶段介入 → L20 last_prompt_token
- ✅ 先ECHO挤压验证回路 → 再训练去噪映射
